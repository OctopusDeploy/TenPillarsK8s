Maintainable deployments

Getting your deployments to the production environment is just the beginning. Diagnosing issues, collecting logs, performing backups, restarting services, rotating keys and testing connections are just some of the day to day operations tasks that keep your applications running and your customers happy.

While you could SSH or RDP into a server and start poking around, or execute a few configuration changes from your local PC, each change you make drifts your production environment from your non-production environments, making it just that little bit harder to implement repeatable deployments. It is also difficult to track the changes that were made, verify that they worked, and audit who changed what.

Like deployments, maintenance tasks should be repeatable, verifiable, visible, measurable, auditable, standardized and coordinated. Maintenance tasks represent the business knowledge required to keep your deployments running, and should be held to the same standards as your deployment processes.

Throughout this book we have been switching from deployment processes, which we use to promote Kubernetes resources with any associated scripts through environments, and runbooks, which we use to query the cluster and run administration tasks in any environment.

This distinction may seem a little arbitrary at this point. After all, runbooks and deployment processes run exactly the same steps, and a lifecycle like Hotfix allows a deployment to be made to any environment in any order. In fact, before the release of runbooks in Octopus, management tasks were frequently modeled as deployment processes with permissive lifecycles.

However, as your deployment infrastructure grows, the delineation deployment processes and management tasks become more apparent:

  • Deployments tend to be executed in response to new code or configuration being developed, while runbooks will often be run on a more ad-hoc basis to query information or respond to unexpected events.
  • Deployments follow a strict progression through environments, while runbooks need to be run in any environment at any time.
  • Deployments should progress a static release through environments, while the latest stable runbook should be run without concerns for environment progression.

Without calling much attention to it, we have been gradually building a library of runbooks to ensure the manageability of our deployments long after the original deployment process finished. From the very beginning we modelled the creation of our targets through runbooks like Create Frontend Targets and Create Backend Targets. We frequently referred to query runbooks like Get Service IP. The workers placed in the cluster we implemented in runbooks like Create Kubernetes Worker.

What we never did in this book were calls to kubectl from a local machine. This was a very deliberate decision, and now we reap the benefits.

Our library of runbooks provides us with a robust selection of management scripts that can be handed over to a new team, and as a reference library for anyone looking to recreate the infrastructure we have created throughout this book. A new member of your team can begin maintaining deployments with nothing more than a web browser and a Octopus login - there is no need to hand out cluster credentials, install local tools like kubectl, or generate configuration files.

Just as importantly, by using runbooks we get so many of the other pillars for free: each runbook can be run repeatably across environments, the results of a runbook run is visible in the Octopus dashboard, each runbook can be tested and verified before handed to a team, Octopus will audit important runbook events, runbooks can be standardized with step templates or copied as part of an import/export process, and (as well see in the next chapter) run books can be coordinated like deployment processes can.

Getting the most from runbooks

If you know how to create a deployment process, you know almost everything you need to create a runbook. Runbooks do have some specific features that are worth calling out though.

Publishing snapshots

Until now we have simply run runbooks after we create them. Behind the scenes running a runbook in this way creates a snapshot of the variables and packages, much as we do when creating a release for deployment. This snapshot is called a draft.

Continually running runbooks in this way allows you to tweak and verify the process, eventually reaching a point where the configuration is considered stable. At this point a snapshot can be published to create a blessed snapshot intended for general use. By clicking the PUBLISH button in the runbook editor presents a page where the package versions and variables can be defined. Clicking the PUBLISH button on the results of a snapshot runbook run will capture the current packages and variables in the published snapshot.

When a published snapshot is available we have the option to run either the published or draft snapshot when running a runbook. The published snapshot is the best choice for day to day maintenance tasks.

Limiting environments

The default configuration presents all environments in the space to run a runbook in. However, it often only makes sense to run runbooks in a subset of the available environments.

We've already created some runbooks that are limited to the Admin environment by clicking the SETTINGS tab, selecting the Run only in specific environments option, and defining one or more environments.

Where runbooks are collocated with deployment projects, it may make sense to limit the environments a runbook can run it to those defined in the deployment project lifecycles. This is done by selecting the Environments from Project Lifecycles option.

Variable prompts and scoping

We've used variables previous as a way of exposing different values based on the environment being deployed to, or to hide secret values. Another use case that is especially relevant to runbooks is the ability to prompt the end user for a variable value with each deployment or runbook run.

Conclusion

An application's lifecycle extends far beyond a successful deployment to production. Routine maintenance and unexpected events mean we will be interacting with our environments constantly between deployments. In Octopus, runbooks embrace the ad-hoc and unpredictable nature of maintenance and support tasks, while still retaining the benefits of automated deployment processes.

In this chapter we:

  • Discussed the differences between deployment processes and runbooks.
  • Highlighted the benefits we gained through the disciplined used of runbooks over direct kubectl commands on a local machine.
  • Highlighted some options to make runbooks easier to use.

The next and final pillar we'll present is coordinated deployments where we look at how Octopus can be integrated into external tools and business processes.
